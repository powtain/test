# 项目协作规范

本文档定义本仓库在 `Golang` 与 `Python` 开发中的通用语言规范。`AGENT.md` 与本文件保持一致；若两者存在冲突，以最新提交中两者一致后的内容为准。

## 通用原则

- 优先编写清晰、可维护、可测试的代码，避免过度设计。
- 修改现有逻辑前先理解上下文，尽量复用既有模式，不随意重写稳定代码。
- 命名应直接表达业务意图，避免无意义缩写，允许行业内通用缩写。
- 单个函数应尽量只做一件事；当分支、嵌套或状态过多时主动拆分。
- 注释只解释意图、边界条件或非显而易见的取舍，不重复代码表面含义。
- 新增逻辑默认补充测试；若无法补充，需要在提交说明中解释原因与风险。
- 引入新依赖前先评估必要性、维护状态、许可证与替代方案。

## Golang 规范

### 目录与组织

- 遵循 Go 默认项目布局，优先保持扁平结构；仅在规模增长明显时再引入 `internal/`、`pkg/` 等目录。
- 同一 package 内的文件按职责拆分，避免按“类型名”机械分文件。
- `main` package 只负责组装依赖、读取配置、启动流程；业务逻辑放到可测试函数或独立 package。

### 命名与 API

- 导出标识符使用 PascalCase，非导出标识符使用 camelCase。
- 接口命名优先体现行为，例如 `Reader`、`Store`，避免 `IUserService` 这类前缀。
- receiver 名称简短且稳定，通常使用类型首字母缩写。
- 错误变量使用 `err`，上下文错误使用 `fmt.Errorf("...: %w", err)` 包装。

### 代码风格

- 始终使用 `gofmt` 或 `go fmt ./...` 格式化，不手动对齐空格。
- 导入顺序交给 `gofmt` 或工具处理，不保留未使用 import。
- 尽量使用早返回减少嵌套；对异常路径先处理，再保留主流程清晰展开。
- 谨慎使用 goroutine；只有在并发能明确提升吞吐或隔离耗时操作时才引入。
- 共享状态必须显式保护，优先通过 channel 传递数据，其次使用 `sync` 原语。

### 错误处理、日志与配置

- 不忽略错误，除非该忽略是有意识且安全的；必要时用注释说明原因。
- 不使用 panic 处理常规业务错误；panic 只用于不可恢复的编程错误或进程启动失败。
- 日志记录要包含定位问题所需的上下文，但不要输出密码、令牌、密钥等敏感信息。
- 配置通过显式结构体承载，提供默认值与校验逻辑，避免在业务代码中散落读取环境变量。

### 依赖与测试

- 使用 `go mod tidy` 保持 `go.mod` 与 `go.sum` 干净。
- 提交前至少运行 `go test ./...`；若项目已引入 lint，再运行约定的 lint 命令。
- 测试优先使用表驱动风格，覆盖正常路径、边界条件和失败路径。
- 对外部依赖使用接口或可替换组件隔离，避免测试直接依赖真实网络、文件系统或时间。

## Python 规范

### 目录与组织

- 默认采用 `src/` 布局或清晰的顶层包目录，避免把业务代码散落在脚本根目录。
- 模块按领域或职责划分，避免单文件堆积过多类与函数。
- 脚本入口使用 `if __name__ == "__main__":`，业务逻辑保持可导入、可测试。

### 命名与类型

- 遵循 PEP 8：模块、函数、变量使用 `snake_case`，类使用 `PascalCase`，常量使用 `UPPER_SNAKE_CASE`。
- 公共函数、方法与复杂内部函数默认补充类型标注；新增模块尽量做到完整类型提示。
- 对外接口、数据结构和返回值含义不明显时，使用 `TypedDict`、`dataclass` 或清晰的类型别名表达意图。
- 不滥用 `Any`；确需使用时应局部收敛，并在边界处尽快转回具体类型。

### 代码风格

- 使用 `ruff format` 或 `black` 做统一格式化；使用 `ruff check` 做静态检查。
- 导入分组按标准库、第三方、本地模块排列，并交给工具自动整理。
- 优先使用显式异常处理和早返回，避免过长的 `if/else` 链与隐式状态修改。
- 默认使用虚拟环境；依赖通过 `pyproject.toml` 管理，避免手写未锁定的临时依赖列表。
- 不在库代码里直接打印调试信息；需要输出时统一走日志模块。

### 错误处理、日志与配置

- 不吞异常；捕获异常时只捕获明确类型，并补充必要上下文后重新抛出或转换。
- 面向用户的错误信息应清晰可操作，面向排障的日志应保留关键上下文。
- 配置集中管理，可使用环境变量、配置文件或设置对象，但必须有默认值、校验和来源说明。
- 严禁在代码、测试和示例中提交真实密钥、令牌或含敏感信息的数据文件。

### 依赖与测试

- 默认使用 `pytest` 进行测试；测试文件命名使用 `test_*.py`。
- 新增功能至少覆盖一个成功路径和一个失败或边界路径。
- 涉及时间、随机数、网络、数据库、文件系统时，优先使用 fixture、mock 或工厂隔离副作用。
- 提交前至少运行项目约定的格式化、lint 和测试命令；如果仓库尚未建立 Python 工具链，需在变更说明中明确这一点。

## 提交前最低检查

- `Golang` 改动至少执行：`gofmt -w .`（或针对改动文件）、`go test ./...`
- `Python` 改动至少执行：`ruff format` 或 `black`、`ruff check`、`pytest`
- 确认新增依赖、配置项和命令已在文档或变更说明中交代清楚。
- 确认 `CLAUDE.md` 与 `AGENT.md` 内容保持一致。
